1. Executive Summary
   DriveLink is a full-stack web application enabling customers to rent or purchase vehicles through a clean, role-separated interface. The platform integrates Paystack for real payment processing, a smart ordering engine for intelligent car recommendations, and a complete admin control panel for fleet and booking management.

The system distinguishes rental inventory from purchase inventory вЂ” cars are categorized independently for each transaction type вЂ” and enforces stock integrity through atomic database operations. All financial transactions flow through Paystack's verified webhook pipeline, ensuring payment confirmation before any booking or purchase is activated.

5. Functional Requirements
   5.1 Authentication System
   5.1.1 User Registration
   вЂў Fields: first name, last name, email, phone number, password, confirm password
   вЂў Email must be unique вЂ” validated at model and serializer level
   вЂў Password hashed with Django's default PBKDF2 algorithm
   вЂў On success: JWT access + refresh tokens returned, user redirected to /dashboard

5.1.2 User Login (/login)
вЂў Accepts email + password
вЂў Returns JWT access token (15 min expiry) + refresh token (7 days)
вЂў Token payload includes: user_id, email, role, full_name
вЂў Failed login returns 401 with generic error message

5.1.3 Admin Login (/admin/login)
вЂў Separate endpoint: POST /api/auth/admin/login/
вЂў Only accounts with role=admin can authenticate here
вЂў Non-admin credentials return 403 Forbidden

5.1.4 Token Refresh
вЂў Axios interceptor auto-refreshes on 401 response
вЂў Refresh token stored in httpOnly cookie (security) or localStorage
вЂў On refresh failure: user is logged out and redirected to login

5.2 Car Management (Admin)
5.2.1 Add Car
вЂў Fields: name, brand, model year, description, category (rental/purchase), fuel type, transmission, seats, color, daily_rate (rental only), purchase_price (purchase only), stock, main image, gallery images (up to 5)
вЂў Category is mutually exclusive вЂ” a car is either for rent OR for purchase
вЂў Image upload via multipart/form-data, stored in MEDIA_ROOT
вЂў Stock must be >= 1 on creation

5.2.2 Edit Car
вЂў All fields editable post-creation
вЂў Image replacement supported вЂ” old image deleted from filesystem
вЂў Stock can be increased (restocking) but not reduced below current active bookings count

5.2.3 Delete Car
вЂў Soft delete вЂ” car is marked inactive, not removed from DB
вЂў Deleted cars hidden from user-facing catalog
вЂў Existing active rentals/purchases referencing this car are preserved

5.3 Car Catalog (User)
5.3.1 Rental Catalog (/cars/rental)
вЂў Displays all active cars with category=rental
вЂў Smart ordering applied by default (Section 5.6)
вЂў Filters: brand, fuel type, transmission, seats, price range, availability
вЂў Search: by name or brand (debounced, 300ms)
вЂў Each card: main image, name, brand, daily rate, availability badge, View Details CTA

5.3.2 Purchase Catalog (/cars/purchase)
вЂў Displays all active cars with category=purchase
вЂў Same smart ordering, filters, and search as rental catalog
вЂў Each card: main image, name, brand, purchase price, stock badge, Buy Now CTA
вЂў Out-of-stock cars displayed with disabled CTA and greyed badge

5.3.3 Car Detail Page (/cars/:id)
вЂў Gallery carousel, full specifications table, description
вЂў Rental cars: date range picker (start date, end date), total cost auto-calculated
вЂў Purchase cars: quantity selector (max = current stock), total cost displayed
вЂў 'Proceed to Payment' CTA triggers Paystack checkout

5.4 Rental Flow

1. User selects a rental car and chooses start/end dates
2. System validates: car available, dates not overlapping existing bookings, start >= today
3. Booking record created with status=PENDING
4. Paystack transaction initialized вЂ” reference stored on booking
5. User completes Paystack payment (redirect or inline popup)
6. Paystack webhook hits POST /api/payments/webhook/ вЂ” signature verified
7. On charge.success: booking status set to CONFIRMED, car availability updated
8. On failure: booking status set to FAILED, no inventory change
9. User sees confirmation page with booking details and reference

5.5 Purchase Flow 10. User selects a purchase car and quantity 11. System validates: stock >= quantity requested 12. Purchase record created with status=PENDING, stock NOT yet reduced 13. Paystack transaction initialized 14. User completes payment 15. Webhook receives charge.success вЂ” signature verified 16. Stock atomically reduced: Car.objects.select_for_update() вЂ” quantity subtracted 17. Purchase status set to CONFIRMED 18. If stock reaches 0: car marked unavailable automatically 19. User receives purchase confirmation with receipt details

5.6 Smart Ordering Engine
The smart ordering engine ranks cars in the catalog using a weighted scoring algorithm computed at query time. It is not ML вЂ” it is a deterministic, configurable scoring function applied per car.

Signal Weight Description
Booking Popularity 35% Number of confirmed bookings/purchases in last 30 days
Availability 25% Cars with stock > 3 scored higher; unavailable cars penalized
Price Match 20% Proximity to user's session-inferred budget range
Recency 10% Newer listings scored higher (added_at recency)
Rating / Reviews 10% Average user rating (if review system active, else neutral)

вЂў Score computed server-side in CarListView вЂ” annotated via Django ORM aggregate
вЂў Sort order: DESC by score. User can override with manual sort (price asc/desc, newest)
вЂў Budget inference: derived from user's last 3 completed transactions, defaulting to mid-range

5.7 Paystack Integration
5.7.1 Initialize Payment
вЂў Endpoint: POST /api/payments/initialize/
вЂў Payload: amount (kobo), email, reference (UUID), metadata {booking_id or purchase_id, type}
вЂў Returns: authorization_url for redirect or inline embedding
вЂў Reference stored on Booking/Purchase before redirect

5.7.2 Verify Payment (Webhook)
вЂў Endpoint: POST /api/payments/webhook/
вЂў HMAC-SHA512 signature verified using Paystack secret key before any processing
вЂў Invalid signature returns 400 immediately вЂ” no DB changes
вЂў Handles events: charge.success, charge.failed, transfer.success
вЂў Idempotent: duplicate webhook calls for same reference are no-ops

5.7.3 Frontend Redirect
вЂў After Paystack redirect, frontend hits GET /api/payments/verify/?reference=xxx
вЂў Returns booking/purchase status вЂ” frontend shows success or failure UI

5.8 History & Dashboard (User)
вЂў GET /api/rentals/my/ вЂ” paginated list of user's rentals (status, dates, car, amount)
вЂў GET /api/purchases/my/ вЂ” paginated list of user's purchases (car, quantity, amount, date)
вЂў Filter by status: all, confirmed, pending, failed
вЂў Each entry: expandable detail row with Paystack reference and receipt download link

5.9 Admin Dashboard
вЂў Summary cards: total cars, active rentals, total purchases, revenue (sum of confirmed payments)
вЂў Charts: bookings by month (last 6 months), revenue trend, top 5 cars by bookings
вЂў Recent bookings table: user, car, type, amount, status, date
вЂў Quick action links: Add Car, View All Bookings, Manage Users

10. Smart Ordering вЂ” Implementation Detail
    The ordering engine is implemented as an annotated queryset in CarListView. No external ML library is required вЂ” pure Django ORM aggregates produce a numeric score per car.

10.1 Score Formula
score = (0.35 _ popularity_score) + (0.25 _ availability*score) + (0.20 * price*match_score) + (0.10 * recency_score) + (0.10 \* rating_score)

вЂў popularity*score: Count of CONFIRMED rentals/purchases in last 30 days, normalized 0вЂ“100
вЂў availability_score: 100 if stock > 3, 50 if stock 1вЂ“3, 0 if unavailable
вЂў price_match_score: 100 - abs(car_price - user_budget) / user_budget * 100, floored at 0
вЂў recency*score: Days since added, scaled: (30 - min(days_old, 30)) / 30 * 100
вЂў rating_score: avg_rating / 5 \* 100 if reviews exist, else 50 (neutral)

10.2 Budget Inference
вЂў If user has >= 1 prior confirmed transaction: budget = avg(total_amount) of last 3
вЂў Else: budget defaults to median price across all active cars in that category
вЂў Budget passed as context to scoring function вЂ” not stored on user model

11. Security Considerations
    Concern Mitigation
    JWT Secret Exposure SECRET_KEY in .env, never committed. Access tokens short-lived (15 min).
    Paystack Webhook Forgery HMAC-SHA512 signature verified on every webhook before processing
    Stock Race Conditions select_for_update() used in purchase confirmation вЂ” atomic stock decrement
    Rental Date Overlap DB-level validation + queryset overlap check before booking creation
    Admin Route Leakage role field checked server-side on every admin-gated endpoint вЂ” not just frontend guard
    Media File Access Sensitive files served via Django вЂ” not exposed directly through Nginx in dev
    SQL Injection Django ORM used exclusively вЂ” no raw queries
